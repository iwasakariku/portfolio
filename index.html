-0650M / MESOPELAGIC ZONE

止められないシステムの、止まらない設計。

システムが止まると物流が止まる、ミッションクリティカルな業務システムを主戦場にしてきた。 深く潜るほど、問題の根はシステムより下にある。 トランザクションの競合、デッドロック、インデックス設計の歪み—— 表層からは見えない場所で起きている事故は、そこまで潜らないと直せない。

技術的正しさだけでは設計は決まらない、とここで学んだ。

Fit to Standard。

顧客ごとのカスタマイズ要求に都度応えるのではなく、業務を標準に寄せる導入設計。 製造・小売・C2Cなど異なるドメインの大手企業への導入を重ねる中で、 「作らない判断」の価値を体に入れた。

オニオンアーキテクチャのSaaS適用。

マルチテナントの分離設計、ドメイン層を汚さない境界の引き方、 リスクを織り込んだ技術選定。責務を分離し、依存を極小化する。 速く変えられるコードは、結局それしかない。

品質は仕組みで担保する。

PestPHP・Playwrightによるテスト整備、CI/CDパイプラインの高度化。 レビューと根性に頼らず、壊れたらパイプラインが止まる状態を作る—— それがテックリードの仕事だと思っている。

本番障害の解析も守備範囲。トランザクションのロック競合、デッドロック、 インデックス設計起因の性能劣化。

  • PHP / Laravel
  • MySQL / InnoDB
  • マルチテナント設計
  • PestPHP / Playwright
  • CI/CD
  • 障害解析

-2200M / ABYSSAL APPROACH

AIに書かせる、ではなく。AIと潜る。

直近の実例。新規プロジェクトにアサインされた直後、オンボーディングと並行して 1週間で約2万行のリファクタリングを終えた。 種明かしは根性ではなく、運用の設計だ。

01

人間の承認ゲートを、計画OKの一語まで圧縮した。

リファクタ用のエージェントスキルを自作。ブランチ作成からコミットまで自動運転。 止めるのは計画確認の一点だけ。push / PR は常に人間が押す。

02

自動化した直後に、自動化を疑う。

未来目線でリスクマップを引き、危険な一点だけを巻き戻す。 全部戻すのではなく、安全圏を定義して運用に乗せる。 自動化の価値はアクセルより、ブレーキの設計で決まる。

03

レビューもAIを重ねる。

実装したAIとは別のAIに壁打ちさせ、対応スコープを「やる・見送り・後回し」に仕分ける。 人間が集中するのは、スモークテストとUXの違和感、 そして「この仕様、そもそも危なくないか」という問いだけ。

しかもこの1週間は、新規プロジェクトにアサインされた直後。 ドメイン知識のキャッチアップ自体もAIにやらせながら、同じ日にその機能を直していた。 オンボーディングとリファクタが同時に走る——これがAI前提の開発フローだ。

対象はコード生成だけじゃない。要件着手からPull Request作成、 振り返りを通じたAIのコンテキスト更新まで。SDLC全体を、 AIエージェントが連続的・自律的に走れる環境として設計している。 使い捨てのコード生成で終わらせないこと。それがPoCと実運用の分かれ目だ。
