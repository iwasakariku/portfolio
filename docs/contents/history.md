4年目

アプリ/サーバーAPI、管理画面の開発

【概要】0→1の新規プロジェクトから保守案件まで、大手クライアント案件の開発および上流工程を担当。

1.マルチプラットフォーム開発と技術選定： Android(Kotlin)を用いたネイティブ開発、サブスクリプション設計、FirebaseによるPush通知基盤の構築に従事。自ら技術調査を行い「テナント自動切替ライブラリ」を導入することで、効率的なSaaS構築を実現。

2.上流工程からの一気通貫な関与： カスタマーサクセスの経験を活かし、顧客との直接折衝から参画。曖昧な要求を要件・仕様・設計へと落とし込み、ビジネス視点を持った開発ディレクションを遂行。

3.主要実績：農業xマッチングアプリ（大手人材系）、スポーツ教育x動画解析アプリ（大手通信キャリア）、こども向け動画事業（大手教育系）等のバックエンドおよびモバイル開発。

アプリ向けAPI開発（組織連携・プロフィール機能）

【概要】元上司（PM）からの要請により、大手通信キャリア提供のスポーツ教育x動画解析アプリの機能拡張を担当。

1.制約下における高度な設計能力：「アプリ側に個人情報を保持できない」というセキュリティ制約のもと、既存インターフェースを維持しつつ効率的なデータ成形・連携を実現。保守性と整合性を両立させたAPI設計を完遂。

2.スケーラビリティを考慮した実装：将来的なトラフィック増を見据え、組織登録時のデータリレー処理における整合性担保と負荷分散を意識した処理フローを構築。

3.自律的なプロジェクト推進：PMとダイレクトに連携し、曖昧な仕様を早期に明確化。手戻りを最小限に抑え、本業と並行しながら高品質なプロダクト提供を実現。

5年目

物流関連

1.秒間8決済（1分間約500件）規模のトランザクションが発生する外部サービスとのAPI連携において、自社DBのデッドロック回避とデータ整合性の担保が至上命題であった。

2.排他制御（悲観ロック/楽観ロック）の適切な使い分けと、トランザクションの局所化により、同時実行性を最大化。

3.APIレスポンスの高速化のため、EloquentのN+1問題の完全排除に加え、MySQLのインデックス設計をクエリプランに基づいて最適化。

4.データプロバイダーによるパターン網羅や境界値分析に基づき、PestPHPでカバレッジ95%を維持。複雑なビジネスロジックに対しても、自動テストによる品質ガードレールを構築することで、リリース後の不具合0件を継続し、開発スピードと安全性を両立させた。

5.成果：性能遅延や不具合による問い合わせ0件の安定稼働を継続。この高い開発品質がクライアントより高く評価され、当初のプロジェクト範囲を超えた別プロジェクト(ブランド返品の鑑定業務WMS導入)の打診・進行に直結。技術力によって自社の事業機会拡大に直接的に貢献した。

6年目

1.開発メンバー30名規模の大型案件において、構成管理責任者として参画。Gitブランチ戦略（Git Flow等）の策定、プルリクエスト（PR）の全件レビュー、およびSTG/本番環境へのデプロイ管理を統括し、大規模開発におけるコード品質と整合性を担保した。

2.プロジェクト途中のPM離脱に際し、現場のハブとして顧客との仕様調整（Hulft連携等）を直接遂行。外部ベンダーに対しては、PRの記述標準化や単体テスト方針の再定義、オンラインMTGでの技術レクチャーを徹底し、開発の迷走を未然に防いだ。

3.7ヶ月間で400回（1日平均2〜3回、最大10回以上）のデプロイを完遂。リリース作業を「特定の人にしかできない作業」から「手順化された業務」へ昇華させ、自分が不在時でも新卒メンバーが安全にデプロイ可能な体制を構築した。

4.成果：複雑なファイル連携と多人数開発が絡む難所において、デプロイ起因の重大障害ゼロを達成。技術的なガバナンスを効かせることで、炎上状態のプロジェクトを安定稼働へと導いた。

1.崩壊危機のチームにおける心理的安全性の構築 育成リソース不足による離職やメンタル不調が相次いでいた5名規模のチームを再建。メンバーとの年齢の近さを活かした「関係重視型リーダーシップ」を軸に、徹底した対話を通じて個々の得意不得意やボトルネックを言語化。障害を一つずつ取り除くことで、疲弊した組織の心理的安全性を回復させた。

2.SL理論（状況適応型リーダーシップ）に基づく個別育成 メンバーの習熟度に応じ、ティーチング（指示型）からコーチング（援助型）へと柔軟にスタイルを変化。タスクのキャッチアップ支援から始め、「自分中心の視点」から「チーム・PM・ステークホルダーへの貢献視点」へのマインドセット転換を促した。

3.成果：離職率の改善と生産性の向上 個別の動機付けを実践することで、高いモチベーションを維持したまま全員が一人前のエンジニアとして自走できる体制を確立。結果として、離職・体調不良者・日報記載漏れをゼロに抑え、マネージャー層の負担軽減とチームアウトプットの最大化を達成した。

1.未知の技術領域へのキャッチアップと資産化 外部ベンダーが構築したC#製のソケット中継サーバーに対し、自ら技術解析を実施。ベンダー提示の仕様を鵜呑みにせず、Windows依存の環境からMac/Docker(Ubuntu)環境への移植・動作検証を行い、環境を選ばないデリバリー手順を確立した。これにより、ロボット連携の仕組みを他案件でも流用可能な「自社資産」として引き継ぐことに成功。

2.AI駆動開発によるドキュメント整備とボトルネック排除 GitHub CopilotやCursorに加え、AIを活用したフローチャート・シーケンス図の自動生成を導入。複雑な通信仕様を視覚化することで、新卒1年目のメンバーが数時間で仕様を把握できる環境を構築し、オンボーディングのコストを劇的に削減した。

3.静的解析導入とCI/CD改善による品質の底上げ 静的解析ツールの導入や開発フローの刷新を行い、個人のスキルに依存しない品質管理体制を構築。開発リーダーとして窓口を一本化し、コードレビューと単体テスト方針の徹底指導により、期日通りの納品と高い保守性を両立させた。

4.成果 属人性を排除した開発環境の提供により、若手・中途メンバー主体のチームでプロジェクトを完遂。単一案件の成功に留まらず、全社的な「AI駆動開発」と「ロボット連携基盤」の標準化に貢献した。

1. ベトナムベンダーのサポート

2025年後半に新たにパートナーシップを結んだ、言語・文化・開発商習慣の異なる外部ベトナムベンダーに対し、すべての開発業務を安全かつ迅速に移譲（オフショアリング移行）するプロセスを開発リーダーとして主導。これまでの自社の開発フロー、アーキテクチャの注意事項、ドメイン知識のトランスファーを徹底するとともに、GitHub CopilotやCursorといった生成AIツールの活用を前提とした「日越共通の開発規約・プロンプトルール」を整備。国境を越えたリモート体制下でも、手戻りを最小限に抑え、高効率・高品質な開発を自走して進められる生産性の高いガバナンス環境を構築しました。

7年目

1. スクラム運営の初期構築

『アジャイルサムライ』を片手に、オフショア拠点を巻き込んだアジャイル未経験チームの運営基盤をゼロから構築。インセプションデッキを作成してプロジェクトの「目的」と「やらないこと（境界線）」を明確にし、チーム内外の期待値を完全同期させました。また、ユーザーストーリーマッピングにより10年分のレガシー機能を分解して最初のバックログを作成し、ストーリーポイントを用いた相対見積もりによって「ベロシティ」を計測する土台を確立。さらに、スプリント開始と同時に「動くソフトウェア」を最速で届けるための開発環境の初期構築を完了させました。

2. プロジェクトの運営

スクラムを本格導入し、1スプリントごとの成果物（ベロシティ）を右肩上がりに増やしていくために、タスクのボトルネック排除、ステークホルダーとの期待値調整、およびCursorのプロンプトルールや共通規約の整備を牽引。

また、オフショア開発の最大の壁であるコミュニケーション工数を解決するため、これまでの「業務ドメイン知識」と「既存コードベース」を学習させた「生成AIベースの仕様・コード探索Q&Aツール」を自作。これにより、これまで1日1時間程度かかっていたベトナム側とのQ&A工数を5分程度に激減させ、仕様の疑問に即答して開発をノンストップで進められる環境へと改善しました。

3. AWS環境、CDの構築

今回新たに作成する環境において、今までチーム内で実現できていなかったGitHub Actionsによるデプロイ自動化（CI/CD）の仕組みを完全新規で構築しました。

開発者がコードをマージするだけで検証環境が最新状態に自動アップデートされるため、手動デプロイ工数を100%削減。テスト・フィードバックのサイクルが劇的に高速化し、これまで以上に圧倒的に開発がしやすいインフラ基盤を実現しました。

1. AIを使ったフルサイクル開発の導入

スピードと品質を両立させるAI駆動開発の次世代標準として、以下のルール整備およびスキル・機構の構築を主導しました。

2.開発プロセスのオーケストレーション構築： 提示された要件定義・仕様をもとに、AIが自律的に「開発プランの策定 ➔ 実装 ➔ 動作検証 ➔ Gitプッシュ・PR（プルリクエスト）作成」までを一気通貫で実行・完結する自動化フロー（オーケストレーション）を構築しました。

3.ハーネスエンジニアリングの徹底による精度向上： AIに丸投げするのではなく、要所要所で人間（エンジニア）のフィードバックや修正指示を適切に注入し、AI自身がコンテキストを理解して自己改善・コード修正を行うためのプロンプト・評価ハーネス（枠組み）を徹底。生成コードのバグ率を最小化しました。

4.DoD（完了定義）を満たす最小限のスモークテスト構築： 今後のアーキテクチャ刷新や大幅なリファクタリングを想定し、テストコードの陳腐化リスクを回避。本番移行時のデグレを確実に防ぎつつ、投資対効果を最大化する「DoD（完了定義）を満たす最小限かつコアなスモークテスト」へとテスト戦略を実施しました。
