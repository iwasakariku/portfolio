-0100M / PART 01

背景：ドメイン・アーキテクチャの理解

1年ほどかけて作られたPoC（概念実証）コード。
大量の古いドキュメントが残されている中、ミーティング（MTG）でプロジェクトの要点だけが口頭で伝えられるような状況でした。
コードや、雑多に書かれたMarkdownファイルの山をただ読むだけのオンボーディングでは、動的な仕様変更や暗黙のルールを理解するのに限界がありました。

そこで、まず「onboarding」ディレクトリを作成し、そこに自分が理解した仕様やコンテキストをドキュメントとして整理することから始めました。 これこそが、リファクタリングタスクを安全に遂行するための一番の近道でした。

「手を動かすより、先に、全体を把握してから進める。」
それは、AIを大規模に使う際にも全く同じ原則が適用されます。

-0200M / PART 02

プロセス：AIの推進力と人間のブレーキ

AIは高速にコードを生成できますが、放置すると「それっぽいけれど境界を跨ぐ依存関係」を作りがちです。 そのため、以下のような明確な協働プロセスを定めました。

まず、AIに「いきなりコードを書き換えさせない」というルールを徹底しました。 最初に変更すべきクラスの依存関係やドメイン構造を分析させ、人間が「リファクタリング計画」として承認します。 また、承認できない計画が提案された場合は、そのプロセスを徹底的に分析してAIにフィードバックします。 そうすることで、同じ過ちを繰り返さないようにプロンプトや前提知識をアップデートします。 これにより、次回以降はより手放しで任せられる高精度なスキルへと進化していきます。 そして、最終的に承認された計画の範囲内でのみ、1モジュールずつコードを書き換えさせました。

PHP (Pest) 
// リファクタリング後の境界をテストで固定する例
test('inventory allocation respects tenant isolation', function () {
    $tenantA = Tenant::factory()->create();
    $tenantB = Tenant::factory()->create();

    // テナントAのコンテキストで処理を実行
    Context::set('tenant_id', $tenantA->id);
    $service = app(AllocationService::class);
    
    $service->allocateStock([
        'sku' => 'ITEM-001',
        'qty' => 10
    ]);

    // 他のテナントのデータベースに影響が出ていないことを検証
    expect(Stock::where('tenant_id', $tenantB->id)->count())->toBe(0);
});

コード変更後は、直ちにテストを実行しました。今回はUIが大きく変わることも想定し、手動によるスモークテストを中心に行いました。 これまでいくつものSaaSシステムを開発してきた経験から、ある程度の属人性は避けられませんが、これまでの勘を活かした観点に基づき、DoD（完了定義）を満たせるようレビューの焦点を絞りました。 ついテストコードを書き足してしまいたくなる衝動に駆られますが、トレードオフを明確にし、流行にとらわれず本質を見失わないアプローチを選択することが重要です。

-0300M / PART 03

結果と教訓：仕組みが担保する安心感

結果として、1週間で計23,400行の不要コードおよび設計の整理を行い、 本番環境でのデプロイ後も不具合はゼロでした。

この成果を支えたのは、AIを活用した高速な開発サイクルを安全に回すための 「間違ったらその場で自ら止める仕組み」 と 「同じ過ちを繰り返さないためのプロセス」 をあらかじめ構築していたからです。 これこそが、AI時代におけるシステム開発の新しいスタンダードになると確信しています。
