-0100M / PART 01

背景：ドメイン・アーキテクチャの理解

1年ほどかけて作られたPoCコード。
古いドキュメントが大量の中、MTGでPJの要点だけ伝えられる。
コードや、雑多に書かれたマークダウンの山をただ読むだけのオンボーディングでは、動的な仕様変更や暗黙のルールを理解するのに限界がありました。

そこで実行したのが、onboardingディレクトリを作り、そこに自分が理解したコンテキストを整理する。 それが、リファクタリングタスク遂行の一番の近道でした。

「手を動かすより、先に、全体を把握してから進める。」
それは、AIを大規模に使う際にも全く同じ原則が適用されます。

-0200M / PART 02

プロセス：AIの推進力と人間のブレーキ

AIは高速にコードを生成できますが、放置すると「それっぽいけれど境界を跨ぐ依存関係」を作りがちです。 そのため、以下のような明確な協働プロセスを定めました。

まず、AIに「いきなりコードを書き換えさせない」こと。 最初に、変更すべきクラスの依存関係やドメイン構造を分析させ、人間が「リファクタリング計画」として承認します。 また承認できない計画を生み出したプロセスを徹底的に観測し、AIにフィードバックします。 そうすることで、同じ過ちを繰り返さないように学習させます。 次回以降はより手放しが効く、高いレベルのスキルと進化します。 承認された計画の範囲内でのみ、1モジュールずつコードを書き換えさせます。

PHP (Pest) 
// リファクタリング後の境界をテストで固定する例
test('inventory allocation respects tenant isolation', function () {
    $tenantA = Tenant::factory()->create();
    $tenantB = Tenant::factory()->create();

    // テナントAのコンテキストで処理を実行
    Context::set('tenant_id', $tenantA->id);
    $service = app(AllocationService::class);
    
    $service->allocateStock([
        'sku' => 'ITEM-001',
        'qty' => 10
    ]);

    // 他のテナントのデータベースに影響が出ていないことを検証
    expect(Stock::where('tenant_id', $tenantB->id)->count())->toBe(0);
});

コード変更後は、直ちにテストを実行する。今回はまだまだUIがガッツリ変わることを想定して、手動によるスモークテストでした。 今までいくつものSaasシステムを開発してきた経験から、属人性はありますが、勘を働かせた観点に基づき、DoDを満たすように レビューの焦点を絞りました。 つい、テストコードを書き足してしまいたくなる衝動にかられますが、トレードオフを明確にし流行りにとらわれず 本質を見失わない進め方の選択が重要です。

-0300M / PART 03

結果と教訓：仕組みが担保する安心感

結果として、1週間で計23,400行の不要コードおよび設計の整理を行い、 本番環境でのデプロイ後も不具合はゼロでした。

この成果を支えたのは、AIを活用した高速な開発サイクルを回すための 「間違ったらその場で自ら止める仕組み」 と 「同じ過ちを繰り返さない為のプロセス」 が先に実装していたからです。 これが、AI時代におけるシステム開発の新しいスタンダードになると思いました。
