マルチテナントSaaSにおけるデータ分離設計と境界定義

B2B SaaSの開発において、最もクリティカルな要件の一つが「テナント間の確実なデータ分離」です。 運用コストを抑えつつ、絶対にデータ流出を起こさないための実装設計アプローチについて解説します。

-1000M / PART 01

データ分離のトレードオフ：物理分離 vs 論理分離

マルチテナントにおけるデータ分離の手法は、大きく「物理分離（テナントごとにデータベースを分ける）」と「論理分離（共有データベース内の各レコードにテナントIDを持つ）」に分かれます。

物理分離は、インフラレイヤーで接続が隔離されるためデータ混入リスクが極めて低いというメリットがあります。 しかしその一方で、数千・数万テナントにスケールした際のデータベース管理コストの肥大化、スキーマ一斉アップデートの難易度上昇、および費用対効果の面で大きな課題を抱えることになります。

物流現場で使われるB2B SaaSのように、大量のトランザクションが走るシステムでは、スキーマ変更の柔軟性とコスト最適化を考慮し、共有DBを用いた「論理分離」を採用しつつ、いかにアプリケーション境界でミスを防ぐかが鍵になります。

-1200M / PART 02

境界の実装：Laravel グローバルスコープの活用

論理分離における最大の脅威は、「SQLのWHERE句に `tenant_id` を書き忘れる」という人為的ミスです。 これを防ぐため、フレームワークレベルでクエリ発行時に自動でフィルタリングが適用される「グローバルスコープ」を構築します。

PHP (Laravel) 
trait BelongsToTenant
{
    public static function bootBelongsToTenant()
    {
        // モデル作成時に自動でカレントテナントIDを設定
        static::creating(function ($model) {
            if (!$model->tenant_id) {
                $model->tenant_id = CurrentTenant::id();
            }
        });

        // クエリ検索時に自動で tenant_id フィルタを付与
        static::addGlobalScope(new TenantScope);
    }
}

このトレイトを各ドメインモデルに付与することで、開発者がSQLを意識せずとも、自動的に所属テナントのデータだけにアクセスできるガードレールが形成されます。

TENANT ISOLATION WORKFLOW
Client HTTP Request
TenantMiddleware (Set Current Tenant ID)
TenantScope (Appends Tenant ID to SQL WHERE)
Database query (Filtered & Secure)

-1500M / PART 03

二重のセキュリティ：インフラと自動検証

しかし、グローバルスコープだけに頼るのも危険です。 たとえば、「スコープ無効化メソッド（`withoutGlobalScopes`）」を特定の管理者向け機能で使用した際に、意図せず他テナントへのデータ漏洩を引き起こす脆弱性が作り込まれてしまう可能性があります。

そのため、二重の防御ラインを設計します。

1つ目は、コントローラーやポリシー層での明示的なアクセス制御（ゲートキーパー）。 2つ目は、CI/CDで必ず回る「テナント間クロスアクセス制限テスト」です。 ダミーテナントを複数作成し、テナントAの認証トークンを用いたAPIリクエストが、絶対にテナントBのデータを参照・変更できず、一律で403/404エラーになることを自動テストで徹底的に保証しています。

「数年後の自分が泣かないために、依存を最小化し、変えやすいコードを書き続ける」。 マルチテナント設計は、まさにその思想が試されるコア領域と言えます。
