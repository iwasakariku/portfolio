-2000M / PART 01

デッドロック発生：高並行の臨界点

物流の出荷指示データは膨大です。荷主からのデータ送信や倉庫からの出荷確定が、ある時間帯に一斉にトランザクションとして走ります。 ある日、特定商品の在庫引き当て（Allocation）を行う処理で、デッドロックエラーが本番環境で頻発するようになりました。

不具合件数そのものは数%でしたが、物流が止まりかねないコアの引き当て処理であるため、最優先での原因究明が必要となりました。 アプリケーションのコードを見ただけでは競合箇所がわからず、データベース内部の「ロックの取得順序」まで深く潜る必要がありました。

「表面からは見えない、システム境界の歪みと向き合う。」
これこそが、サーバーサイドエンジニアの主戦場です。

-2500M / PART 02

ログ解読：InnoDBのロック挙動（Gap Lock, Next-key Lock）

原因を特定するため、MySQLで `SHOW ENGINE INNODB STATUS` コマンドを実行し、最新のデッドロックログを抽出しました。 ログの中身を辿ると、一見関係なさそうな2つの別レコードに対する `SELECT ... FOR UPDATE` と `INSERT` が、お互いにロック解除を待ち合っている状態でした。

原因は、検索クエリがテーブルをフルスキャンした結果、存在しない範囲（ギャップ）にまでロックが及ぶ「Gap Lock（ギャップロック）」および「Next-Key Lock」でした。

SQL 
-- デッドロックの引き金となった範囲検索のロッククエリ
SELECT * FROM stock_allocations 
WHERE tenant_id = 5 
  AND status = 'pending' 
  AND sku = 'SKU-882' 
FOR UPDATE;

このクエリを実行した際、(tenant_id, status, sku) に適切な複合インデックスが存在しなかったため、MySQLはテーブルスキャンに近いロック範囲を選択し、並行実行された別のインサートクエリ（挿入インテンションロック）とデッドロックを引き起こしていました。

TRANSACTION DEADLOCK RELATION

TRANSACTION A

SELECT ... FOR UPDATE
(Requires Gap Lock on SKU range)

MUTUAL WAIT

TRANSACTION B

INSERT INTO stock_allocations
(Blocked by Transaction A's Gap Lock)

-2800M / PART 03

解決策：インデックス追加と更新順序の統制

対応として以下の2つのチューニングを行いました。

1. **複合インデックスの作成**
(tenant_id, status, sku) の複合インデックスを追加しました。 これにより、`SELECT ... FOR UPDATE` がスキャンする範囲を厳密に対象の「特定の1レコード」に絞り込み、不要なギャップロックの発生を完全に抑え込みました。

2. **更新クエリの発行順序の統一**
プログラム内でデータベースを更新する際、テーブルアクセス順序が「親テーブル（Orders）→ 子テーブル（OrderItems）」となるようトランザクション処理を一貫させ、ロック競合の発生確率を物理的に最小化しました。

この変更をデプロイした結果、本番環境でのデッドロック発生件数は完全にゼロ（0件）となり、高負荷時間帯の決済詰まりも解消されました。 「根性を仕組み（今回はインデックスとトランザクション規約）に置き換える」。 深い知識に基づいたシンプルな解決策こそが、本質的な最適化であると証明できた事例です。
